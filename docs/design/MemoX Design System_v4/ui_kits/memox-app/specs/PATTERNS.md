## 1. Entry / exit conditions (per pattern)

| Screen (spec) | Entered from | Preconditions | Exits to | Exit triggers |
| --- | --- | --- | --- | --- |
| `dashboard` | app launch / Today tab | authenticated or local session | study session, library, create sheet | tab change; Start review; FAB |
| `library` | Library tab | — | deck detail, create sheet, search, selection | row tap; FAB; back exits mode first |
| `flashcard-editor` | card row / add-card / import review | a target deck exists | list (saved) or list (discarded) | Save (valid); back/cancel → dirty guard |
| `study-session` | mode picker / Start review / Continue | ≥1 due or selected card | study-result; dashboard (exit) | finish last stage; exit (X) → confirm |
| `study-result` | end of a study session | a completed session record | dashboard, next session | Continue studying; Done; retry on finalize error |
| `import` | library / deck menu → Import | a target deck | deck (done) or discard | Import success → done; back/cancel → confirm if data entered |
| `export` | deck/library menu → Export | ≥1 deck/card selected | share sheet / done | Export complete; back |
| `search` | app-bar/search dock | — | result detail; back to origin | back exits search mode (query preserved per nav rules) |

**Entry/exit contract.** Every pattern names: what must be true to enter (preconditions), the
single primary objective (from the generated spec §1), the valid exits, and what each exit does to
unsaved state (see §2 dirty guard and the navigation & overlays guide §3 for preservation rules).

## 2. Step-by-step flow, incl. back / cancel / retry

**Generic form flow** (`flashcard-editor`, `import`, `export`, settings forms):

1. Enter → fields in initial state (`create`/`edit`, `source`). Primary CTA disabled until valid.
2. Edit → inline validation on blur; submit-time validation on the primary CTA (see decision
   tables). Errors announced via `role="alert"`.
3. Submit → `submitting` (CTA shows progress + disabled); success → `submit-success`/`done` and
   exit; failure → `submit-error` with a **Retry** affordance that re-runs submit without losing
   input.
4. **Back / Cancel** → if the form is dirty, raise the discard-confirm (`_shared/ConfirmDialog`,
   focus returns to the form on cancel); if clean, exit immediately.
5. **Retry** → available on any `*-error` state; re-attempts the same operation; repeated failure
   keeps the entered data and the error visible.

**Focused study flow** (`study-session`): ProgressHeader → answer current stage → Next → next stage;
last stage → `study-result`. **Back / exit (X)** raises the exit-confirm (progress is saved as far as
completed); `resume-error` and `answer-save-error` states offer Retry.

**List / selection flow** (`library`, `flashcard-list`): browse → (optional) enter selection mode →
act via overflow → bulk confirm for destructive actions → outcome (success/partial/failure) →
selection recovery on failure. **Back** exits selection mode before navigating; changing
filter/sort/search drops the current selection (see navigation & overlays guide §3).

## 3. Decision tables for multi-branch screens

### 3a. `study-session` — stage routing & error branches

| Condition | Card queue | Answer save | → State / next |
| --- | --- | --- | --- |
| Session starts, cards due | has due cards | — | `stage1-review` (then 2→5 in order) |
| Card previously failed / lapsed | relearn queue non-empty | — | `relearn` |
| Due-again card surfaced mid-session | due-review triggered | — | `due-review` |
| User taps exit (X) | any | — | exit-confirm dialog; confirm → dashboard, cancel → stay |
| Resume a prior session | prior session load fails | — | `resume-error` → Retry / start fresh |
| Answer submitted | — | save fails | `answer-save-error` → Retry (answer preserved) |
| Answer submitted | — | save ok, more stages | advance to next stage |
| Answer submitted | last stage complete | save ok | → `study-result` |

### 3b. `import` — source/mapping/dedupe branches

| Condition | → State | Next action |
| --- | --- | --- |
| Screen entered | `source` | choose file or paste text |
| Source parsed, columns ambiguous | `mapping` | user maps term/meaning columns |
| Source parsed, mapping known | `preview` | review parsed rows |
| Preview contains rows matching existing cards | `dup-warning` | choose skip / merge / import-anyway, then Import |
| Import running | `importing` | progress; cancel returns to `preview` |
| Import fails (parse/write error) | `import-error` | Retry (keeps source + mapping) |
| Import succeeds | `done` | back to deck; summary of N imported / M skipped |

### 3c. `study-result` — outcome routing

| Condition | → State | Primary next |
| --- | --- | --- |
| Session finished, goal met | `goal-met` | Continue studying / Done |
| Session finished, goal missed | `goal-missed` | Continue studying |
| Many wrong answers | `many-wrong` | Review missed / Continue |
| Finalizing the session record | `finalizing` | (wait) |
| Finalize failed | `finalize-error` → `retry-finalize` | Retry |

(Other multi-state screens — `library`, `flashcard-editor`, the study game modes — follow the
generic flows in §2; their branch states are enumerated in each generated spec's state matrix.)

## 4. Adaptation cross-links (responsive / platform / localization / accessibility)

These concerns are documented once and referenced from every pattern rather than duplicated per
screen:

| Concern | Where it's specified | Applies to every pattern as |
| --- | --- | --- |
| Responsive / viewport / safe-area | `guidelines/` (viewport scope) + `mobile-ui-construction-contract.md` | phone-portrait 390×780 baseline; body scrolls, bars fixed; honour top/bottom safe-area |
| Platform (RN) presentation | each component `*.prompt.md` (RN note) + `readme.md` | one visual language across iOS/Android; sheet/dialog/menu chosen by content, not platform |
| Localization / i18n | `guidelines/i18n-localization.md` | externalized strings, locale date/number/plural formatting, +30–50% expansion, per-script fonts, RTL logical properties |
| Overlays / navigation / state preservation | `guidelines/navigation-overlays.md` | one dismissable layer at a time; focus restore to trigger; scroll/search/filter preserve vs reset |
| Keyboard / reading order / a11y | `guidelines/keyboard-focus-order.md` | DOM = reading = focus order; one heading; announced state changes; visible focus ring |
| Component constraints & combinations | `guidelines/component-constraints-matrix.md` | text/action limits; legal prop combos; nested-interactive rule |

**Per-pattern accessibility notes.**
- Forms (`flashcard-editor`, `import`, `export`): validation errors use `role="alert"`; the submit
  lifecycle announces `submitting`/success/failure (`aria-busy`, `role="status"`).
- Study flow: progress and stage transitions announced; exit-confirm traps and restores focus.
- Lists/selection: selection count uses `role="status"`; bulk destructive confirms state the count
  (localised plural — see i18n guide §5).
